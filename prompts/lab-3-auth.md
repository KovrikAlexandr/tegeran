# Этап 3 — Интеграция Keycloak и полноценная аутентификация backend

## Важно

Данный этап **уточняет исходный общий план** из `prompts/global.md`.

Изначально в `prompts/global.md` этапы были такими:

1. Инфраструктура и база данных
2. Backend (REST API + бизнес-логика)
3. GraphQL-BFF
4. Интеграция аутентификации (Keycloak)

Для текущей реализации порядок **изменён**:

1. Инфраструктура и база данных
2. Backend (core + REST API)
3. **Интеграция аутентификации (Keycloak)**
4. GraphQL-BFF

При выполнении текущей задачи нужно считать, что **интеграция Keycloak выполняется раньше GraphQL-BFF**, потому что BFF должен работать поверх уже аутентифицированного backend.

---

## Цель

Интегрировать Keycloak в существующую систему так, чтобы:

- Keycloak поднимался через корневой `docker-compose.yml`
- Keycloak использовал PostgreSQL
- backend выполнял реальную аутентификацию через JWT, выпущенный Keycloak
- в backend появились endpoint'ы для регистрации и логина
- были добавлены интеграционные тесты основных auth-сценариев

## Архитектурные правила

### 1. Keycloak — источник identity и credentials
Keycloak отвечает за:
- регистрацию пользователя
- хранение credentials
- login
- выпуск access token

### 2. Backend — источник доменной авторизации
Backend отвечает за:
- валидацию токена
- извлечение `sub`
- поиск/создание локального пользователя
- применение бизнес-правил через `ChatMember`

### 3. GraphQL-BFF не реализуется
На этом этапе GraphQL-BFF реализовывать нельзя.

## Backend: аутентификация

Нужно заменить текущую упрощённую auth abstraction на реальную JWT-аутентификацию через Keycloak.

### Backend должен уметь:
- принимать `Authorization: Bearer <token>`
- валидировать JWT
- проверять issuer
- извлекать `sub`
- формировать `CurrentUser`
- использовать `sub` для сопоставления с локальным `User`

### Требования:
- backend не должен хранить пароль пользователя
- `passwordHash` не добавлять
- identity должна идти через `authSubject`
- service слой не должен зависеть от Keycloak SDK или raw JWT

## Feature flag

Существующий feature flag нужно сохранить.

### Переменная:
```env
AUTH_ENABLED=true|false
````

### Поведение:

#### Если `AUTH_ENABLED=false`

* backend работает в mock/development режиме, как и раньше
* protected endpoints доступны без реальной auth
* используется mock user

#### Если `AUTH_ENABLED=true`

* backend требует валидный JWT от Keycloak
* protected endpoints недоступны без токена

## Регистрация и логин

На этом этапе нужно добавить endpoint'ы для регистрации и логина.

Важно: backend не должен самостоятельно реализовывать password auth.
Эти endpoint'ы должны быть **тонким integration layer** поверх Keycloak.

### Нужно реализовать:

#### POST /api/v1/auth/register

Назначение:

* зарегистрировать пользователя через Keycloak
* при необходимости создать/синхронизировать локальный профиль backend

Request:

```json
{
  "name": "Alice",
  "email": "alice@example.com",
  "password": "secret123"
}
```

Ожидаемое поведение:

* пользователь создаётся в Keycloak
* credentials сохраняются в Keycloak
* backend при необходимости создаёт локального `User` с `authSubject`
* возвращается успешный результат

#### POST /api/v1/auth/login

Назначение:

* выполнить login через Keycloak
* получить access token
* вернуть его клиенту

Request:

```json
{
  "email": "alice@example.com",
  "password": "secret123"
}
```

Response:

```json
{
  "accessToken": "...",
  "tokenType": "Bearer",
  "expiresIn": 300
}
```

### Ограничения:

* не реализовывать собственную password storage
* не делать local login вне Keycloak
* backend выступает как integration facade, а не как auth provider

## Локальный пользователь backend

Нужно определить и реализовать стратегию связи Keycloak user ↔ backend user.

Рекомендуемая стратегия:

1. при успешной регистрации через backend:

   * создать пользователя в Keycloak
   * создать локального пользователя в backend с `authSubject`

2. при обращении с валидным токеном:

   * извлечь `sub`
   * найти локального пользователя по `authSubject`
   * если не найден, можно:

     * либо создать автоматически
     * либо вернуть ошибку

Нужно выбрать один вариант и описать его явно в результате.

Предпочтительно:

* **создавать локального пользователя при регистрации**
* при authenticated запросе только находить уже существующего пользователя

## Интеграционные тесты

Тесты должны проверять как минимум:

### Регистрация

* успешная регистрация нового пользователя
* ошибка при повторной регистрации с тем же email

### Логин

* успешный логин
* ошибка логина с неверным паролем

### Protected endpoints

* доступ без токена при `AUTH_ENABLED=true` запрещён
* доступ с валидным токеном разрешён
* доступ с невалидным токеном запрещён

### Current user

* `GET /api/v1/me` работает с валидным токеном
* возвращается локальный профиль, связанный с `sub`

### Совместимость feature flag

* при `AUTH_ENABLED=false` API продолжает работать без реального токена

Этап считается завершённым, если:

1. Keycloak поднимается через корневой `docker-compose.yml`
2. Keycloak подключён к PostgreSQL / `keycloak_db`
