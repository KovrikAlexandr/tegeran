# Глобальный контекст проекта

## Описание

Проект: простой текстовый мессенджер с пользователями, личными и групповыми чатами.

Проект разбит на этапы.  
Каждый этап реализуется отдельно и не должен выходить за свои границы.

Текущий файл описывает общий контекст проекта, который должен учитываться при выполнении любого этапа.

---

## Архитектура

Система состоит из следующих компонентов:

- PostgreSQL — хранилище данных
- Backend — основной сервис с бизнес-логикой (REST API)
- GraphQL-BFF — слой адаптации API под клиента
- Keycloak — внешний провайдер аутентификации

### Распределение ответственности

- Keycloak отвечает за аутентификацию и выпуск токенов
- Backend отвечает за бизнес-логику и авторизацию
- GraphQL-BFF не содержит бизнес-логики и работает поверх REST API
- PostgreSQL хранит доменные данные приложения

---

## Деплой

Компоненты:

### PostgreSQL (third-party)
- база данных `messenger_db` — доменные данные приложения
- база данных `keycloak_db` — данные identity и credentials пользователей

### Keycloak (third-party)
- хранит identity и credentials пользователей
- выполняет аутентификацию
- выпускает access token
- выступает доверенным issuer

### Backend
- работает с PostgreSQL
- реализует бизнес-логику
- валидирует JWT токены
- извлекает identity пользователя (`sub`)
- сопоставляет пользователя с доменной моделью
- выполняет авторизацию на основе `ChatMember`
- предоставляет REST API

### GraphQL-BFF
- не содержит бизнес-логики
- предоставляет GraphQL API
- использует REST API backend
- адаптирует API под клиента
- пробрасывает access token

---

## Monorepo

Проект реализуется как monorepo.

Каждый deployable component должен находиться в собственной директории.

Для каждого деплоймента должна быть своя папка, внутри которой размещаются:
- исходный код приложения
- конфигурация приложения
- `Dockerfile`

Ожидаемая идея структуры:
- `apps/backend` — отдельное NestJS приложение
- `apps/graphql-bff` — отдельное NestJS приложение
- `infrastructure/` — docker-compose, init-скрипты PostgreSQL и общая инфраструктурная конфигурация
- `prompts/` — файлы с глобальным контекстом и промптами этапов

Требования:
- `backend` и `graphql-bff` — это отдельные NestJS проекты
- `backend` и `graphql-bff` не должны быть одним общим NestJS приложением
- у каждого приложения свой `Dockerfile`
- все сервисы поднимаются через общий `docker-compose`

## Функциональные требования

Пользователь:
- регистрируется и аутентифицируется
- работает только со своими чатами
- видит список своих чатов
- отправляет сообщения
- читает историю сообщений
- создаёт и удаляет чаты

### Личный чат (Direct)
- не имеет имени (названия)
- существует между двумя пользователями
- единственный для пары пользователей
- доступен только этим пользователям
- нельзя удалить
- нельзя выйти

### Групповой чат (Group)
- есть имя (название)
- минимум один участник
- создатель — `owner`, остальные — `member`

Участник:
- читает и пишет сообщения
- выходит из чата
- видит список участников

Владелец:
- удаляет чат
- исключает участников

Ограничения:
- ровно один owner
- owner не может выйти из чата

### Сообщения
- не редактируются
- не удаляются

---

## Бизнес-правила

- пользователь работает только со своими чатами
- доступ к чату только у участников
- direct чат уникален для пары пользователей
- group чат имеет ровно одного owner
- роли определяются через ChatMember

## Подход к разработке

Проект реализуется по этапам:

1. Инфраструктура и база данных
2. Backend (REST API + бизнес-логика)
3. GraphQL-BFF
4. Интеграция аутентификации (Keycloak)

Каждый этап:
- является надстройкой над предыдущим
- не должен реализовывать функциональность следующих этапов
- должен оставаться в рамках своего scope

---

## Важные архитектурные ограничения

- Backend — единственный источник бизнес-логики
- GraphQL-BFF не должен дублировать бизнес-логику
- BFF не работает напрямую с базой данных
- Авторизация реализуется только в backend
- Keycloak не содержит доменных ролей `owner/member`
- Prisma считается DAO-слоем backend-приложения

---

## Аутентификация и авторизация

- используется JWT от Keycloak
- токен передаётся через `Authorization` header
- backend валидирует токен
- identity определяется через `sub`
- права доступа проверяются через данные в БД (`ChatMember`)
