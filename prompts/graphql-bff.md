# Этап — GraphQL-BFF

## Важно

Перед выполнением задачи необходимо прочитать:

1. `prompts/global.md`
2. все уже существующие инструкции по backend, auth и REST API
3. `prompts/graphql-bff.md`

Данный этап реализует **отдельный GraphQL-BFF сервис** поверх уже существующего backend.

GraphQL-BFF:
- не содержит бизнес-логики
- не работает напрямую с БД
- не использует Prisma
- не дублирует backend service layer
- работает только через HTTP-вызовы в backend REST API

---

## Цель

Нужно реализовать отдельный GraphQL-BFF сервис в monorepo, который:

1. содержит GraphQL schema
2. содержит HTTP-клиент для backend
3. содержит resolver-ы, использующие этот HTTP-клиент
4. пишет логи в JSON-формате
5. содержит README с подробной инструкцией по ручному тестированию

---

## Архитектурные требования

### 1. Отдельный каталог
GraphQL-BFF должен быть реализован в **отдельной директории** monorepo.

Ожидаемая идея структуры:
- `apps/graphql-bff/`

Внутри должны находиться:
- исходный код приложения
- конфигурация
- `Dockerfile`

### 2. Отдельное приложение
`graphql-bff` — это отдельное NestJS приложение.

### 3. Роль BFF
BFF должен:
- принимать GraphQL запросы
- маппить их на вызовы backend REST API
- пробрасывать access token в backend
- адаптировать API под нужды клиента

BFF не должен:
- реализовывать бизнес-логику
- валидировать доменные правила
- работать с PostgreSQL напрямую
- иметь собственное хранилище доменных данных

## 1. GraphQL schema

Нужно создать GraphQL schema в отдельном `.graphql` файле.

Требования:

- schema должна соответствовать доменной модели проекта
- schema должна покрывать основные пользовательские сценарии
- использовать schema-first подход
- schema должна быть отдельным артефактом, а не генерироваться из TypeScript-классов

### Минимальный состав schema

Нужно предусмотреть:

#### Query
- `health`
- `me`
- `chats`
- `chat(id: ID!)`
- `chatMembers(chatId: ID!)`
- `messages(chatId: ID!, limit: Int, cursor: String)`

#### Mutation
- `register`
- `login`
- `getOrCreateDirectChat`
- `createGroupChat`
- `renameGroupChat`
- `sendMessage`
- `addMemberToChat`
- `removeMemberFromChat`
- `leaveChat`
- `deleteChat`

### Принципы schema
- типы должны быть явными
- input types должны быть явными
- response types должны быть удобны для фронта
- где нужно, использовать `email`, а не `userId`, если backend уже поддерживает это
- схема должна отражать реальные frontend-сценарии

## 2. HTTP client для backend

Нужно написать **отдельный класс HTTP-клиент** для backend.

### Требования

- это должен быть отдельный класс, например `BackendApiClient`
- вся логика REST-вызовов должна находиться только в нём
- одна функция клиента = один вызов backend API
- resolver-ы не должны сами собирать HTTP-запросы
- resolver-ы не должны знать детали URL, headers, query params

### В этом классе должно быть инкапсулировано:
- base URL backend
- отправка HTTP-запросов
- прокидывание `Authorization` header
- обработка ошибок backend
- маппинг ответов backend в удобный вид для resolver-ов

### Примеры методов
Нужно предусмотреть методы примерно такого уровня:

- `health()`
- `getMe(token)`
- `getChats(token)`
- `getChat(chatId, token)`
- `getChatMembers(chatId, token)`
- `getMessages(chatId, limit, cursor, token)`
- `register(payload)`
- `login(payload)`
- `getOrCreateDirectChatByEmail(payload, token)`
- `createGroupChat(payload, token)`
- `renameGroupChat(chatId, payload, token)`
- `sendMessage(chatId, payload, token)`
- `addMemberToChat(chatId, payload, token)`
- `removeMemberFromChatByEmail(chatId, payload, token)`
- `leaveChat(chatId, token)`
- `deleteChat(chatId, token)`

Если в текущем backend какие-то контракты называются немного иначе, нужно аккуратно подстроиться под реальные backend endpoints.

## 3. Resolver-ы

Нужно написать resolver-ы, реализующие GraphQL schema.

### Требования
- resolver-ы должны быть тонкими
- resolver-ы должны вызывать только HTTP client
- resolver-ы не должны содержать бизнес-логику
- resolver-ы не должны напрямую работать с REST через `fetch/axios/httpService`
- resolver-ы не должны знать URL backend

### Что должно быть в resolver-ах
- чтение токена из GraphQL context / headers
- передача токена в HTTP client
- вызов соответствующего метода клиента
- возврат данных в формате GraphQL schema

---

## 4. Аутентификация в BFF

BFF не реализует собственную аутентификацию.

Нужно:

- принимать `Authorization: Bearer <token>` от клиента
- передавать этот токен в backend
- не валидировать JWT самостоятельно, если в этом нет необходимости
- считать backend источником истины по авторизации

Допускается:
- минимальный helper для извлечения bearer token из заголовка

Не нужно:
- подключать Keycloak SDK в BFF
- реализовывать login flow внутри BFF отдельно от backend
- дублировать backend auth middleware

---

## 5. Логирование

Нужно сразу добавить логирование в привычном формате.

### Формат логов
Все логи должны быть JSON и содержать поля:

- `timestamp`
- `level`
- `message`

Пример:
```json
{
  "timestamp": "2026-04-08T12:00:00.000Z",
  "level": "INFO",
  "message": "GraphQL-BFF started"
}
````

### Поддерживаемые уровни

* `DEBUG`
* `INFO`
* `WARNING`
* `ERROR`

### Где логировать

Минимально нужно логировать:

#### INFO

* запуск приложения
* запуск GraphQL сервера
* вызов resolver-а
* успешный вызов backend API

#### WARNING

* отсутствующий или некорректный bearer token
* backend вернул ожидаемую клиентскую ошибку
* частично некорректный входной запрос

#### ERROR

* ошибки HTTP-клиента
* ошибки связи с backend
* неожиданные ошибки resolver-ов
* ошибки старта приложения

#### DEBUG

* технические детали запросов к backend
* маппинг query/mutation → backend endpoint
* параметры пагинации

### Требования

* использовать единый механизм логирования
* не размазывать `console.log` по проекту
* логи должны идти в stdout и быть пригодны для запуска в Docker
