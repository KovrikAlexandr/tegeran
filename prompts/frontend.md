# Этап — Frontend

## Важно

Перед выполнением задачи необходимо прочитать:

1. `prompts/global.md`
2. все существующие инструкции по backend, auth и GraphQL-BFF
3. `prompts/frontend.md`

Данный этап реализует **отдельный frontend-сервис** поверх уже существующего GraphQL-BFF.

Frontend:
- работает через GraphQL-BFF
- не обращается напрямую к backend
- не обращается напрямую к БД
- не содержит бизнес-логики backend
- должен быть простым, устойчивым и быстрым в реализации

---

## Цель

Нужно реализовать минимальный, но рабочий frontend для мессенджера, который:

1. работает как отдельное приложение в monorepo
2. использует GraphQL API из BFF-сервиса
3. покрывает основные пользовательские сценарии
4. имеет простую и аккуратную вёрстку
5. закладывает возможность дальнейшего добавления realtime

---

## Архитектурные требования

### 1. Отдельный каталог
Frontend должен быть реализован в **отдельной директории** monorepo.

Ожидаемая идея структуры:
- `apps/frontend/`

Внутри должны находиться:
- исходный код приложения
- конфигурация
- `Dockerfile`

### 2. Отдельное приложение
Frontend должен быть отдельным приложением на **Next.js**.

### 3. Источник данных
Frontend должен использовать только:
- GraphQL-BFF

Нельзя:
- ходить напрямую в backend REST API
- использовать Prisma
- работать напрямую с PostgreSQL
- обращаться к Keycloak напрямую, если уже есть BFF/backend endpoints для auth

---

## Scope этапа

Нужно реализовать минимальный UI со следующими страницами:

1. страница регистрации
2. страница логина
3. страница со списком всех чатов пользователя
4. страница конкретного чата
5. блок с информацией о чате на странице чата

---

## Сценарии, которые должны поддерживаться

### 1. Регистрация
Пользователь:
- открывает страницу регистрации
- вводит имя, email, пароль
- отправляет форму
- после успешной регистрации перенаправляется на страницу логина

### 2. Логин
Пользователь:
- открывает страницу логина
- вводит email и пароль
- получает access token
- после успешного логина попадает на страницу чатов

### 3. Просмотр чатов
Пользователь:
- видит список своих чатов
- видит базовую информацию по каждому чату
- может открыть конкретный чат

### 4. Просмотр конкретного чата
Пользователь:
- открывает чат
- видит сообщения
- видит информацию о чате
- может отправить сообщение

### 5. Информация о чате
На странице чата нужно показывать:
- имя чата
- тип чата
- список участников
- роли участников

---

## Требования к UI

### Общий подход
Нужно сделать:
- простую вёрстку
- аккуратную вёрстку
- без переусложнения
- без тяжёлых UI-библиотек, если в них нет реальной необходимости

Главная цель:
- рабочий и понятный интерфейс
- минимальный риск ошибок
- достаточная аккуратность для демонстрации

### Нельзя
- делать сложный дизайн
- добавлять много анимаций
- делать тяжёлую компонентную систему
- тратить время на pixel-perfect

### Можно
- простую сетку
- базовые отступы
- простые формы
- простой layout
- базовые стили через CSS modules / simple CSS / Tailwind, если это уже удобно и не усложняет проект

---

## Требования к страницам

### 1. Страница регистрации
Нужно сделать страницу:
- с полями `name`, `email`, `password`
- с кнопкой регистрации
- с переходом на логин

После успешной регистрации:
- показать понятный результат
- перенаправить на логин

---

### 2. Страница логина
Нужно сделать страницу:
- с полями `email`, `password`
- с кнопкой логина

После успешного логина:
- сохранить access token
- перейти на страницу чатов

---

### 3. Страница чатов
Нужно сделать страницу:
- со списком чатов пользователя
- с названием чата
- с типом чата
- с preview последнего сообщения, если оно есть
- с переходом в конкретный чат

---

### 4. Страница конкретного чата
Нужно сделать страницу:
- с сообщениями чата
- с формой отправки сообщения
- с блоком информации о чате

На странице должны быть:
- список сообщений
- поле ввода сообщения
- кнопка отправки
- информация о чате сбоку или отдельным блоком

---

### 5. Блок информации о чате
Нужно показывать:
- имя чата
- тип чата
- список участников
- роли участников

---

## Работа с GraphQL

Frontend должен использовать GraphQL-BFF.

### Требования
Нужно сделать отдельный слой работы с API, например:
- `lib/graphql-client.ts`
- `lib/api/`
- `services/graphql/`

Нельзя:
- размазывать GraphQL запросы хаотично по страницам
- дублировать запросы без необходимости

### Нужно
- вынести GraphQL queries/mutations в отдельное место
- иметь понятный клиент для выполнения запросов
- пробрасывать access token в запросы

---

## Аутентификация на frontend

Нужно реализовать минимальный auth-flow.

### Требования
- после логина сохранять access token
- использовать token для последующих GraphQL-запросов
- если токена нет — защищённые страницы недоступны
- при отсутствии токена можно перенаправлять на `/login`

### Допускается
- хранить token в `localStorage` или в cookie, если это уже удобно
- использовать простой client-side auth guard

### Не нужно
- реализовывать refresh token flow
- делать сложную auth-инфраструктуру
- делать продвинутую security-модель

---

## Закладка под realtime

На этом этапе **realtime реализовывать не нужно**, но нужно заложить возможность добавить его позже без переписывания всего frontend.

### Что нужно сделать
Нужно проектировать код так, чтобы в будущем можно было добавить realtime-подписки.

### Практически это означает
Нужно выделить слой работы с сообщениями, например:
- `message service`
- `chat service`
- `subscribeToChatMessages(chatId, handler)` как заглушку / placeholder / интерфейс расширения

### Сейчас
Достаточно:
- загрузки сообщений по GraphQL
- повторной загрузки по действию пользователя
- при желании можно сделать очень простой polling, только если это не усложняет решение

### Нельзя
- внедрять WebSocket прямо сейчас
- делать Kafka/Redis/realtime инфраструктуру
- усложнять frontend ради будущего realtime

Главное:
- структура кода должна позволять позже добавить механизм подписки на новые сообщения

## Состояние приложения

Нужно выбрать простой подход к state management.

### Предпочтительно
- `useState`
- `useEffect`
- небольшой auth context, если это действительно нужно

### Не нужно
- Redux
- MobX
- сложный event bus
- лишние абстракции

Минимально:
- URL GraphQL-BFF
- порт frontend-приложения

Если в проекте уже принят формат env-конфигурации, нужно ему следовать.

## Docker

У frontend должен быть свой `Dockerfile`.

Если это уже принято в проекте, frontend позже должен быть совместим с запуском через общий `docker-compose`.

На данном этапе главное:
- подготовить frontend как отдельный deployable component

---

## Логирование

Специальное расширенное логирование, как в backend/BFF, на этом этапе не обязательно.

Но нужно:
- не оставлять хаотичные `console.log`
- ошибки запросов показывать пользователю в понятном виде
- технические ошибки обрабатывать аккуратно

---

## Ограничения

Запрещено:

- обращаться напрямую к backend REST API
- обращаться напрямую к БД
- писать бизнес-логику backend на frontend
- реализовывать realtime
- делать сложный UI framework сверх необходимого
- перегружать приложение архитектурой
- пытаться сделать production-grade frontend

---

## Критерии готовности

Этап считается завершённым, если:

1. создано отдельное приложение `apps/frontend`
2. frontend использует GraphQL-BFF
3. реализованы страницы:
   - регистрация
   - логин
   - список чатов
   - конкретный чат
4. на странице чата есть блок с информацией о чате
5. можно:
   - зарегистрироваться
   - залогиниться
   - увидеть свои чаты
   - открыть чат
   - отправить сообщение
6. структура кода позволяет позже добавить realtime
7. вёрстка простая, но аккуратная

---
