# Этап 2.1 — Backend core: DAO + Service + тесты

## Важно

Данный этап **уточняет и временно переопределяет** общий план из `prompts/global.md`.

В `prompts/global.md` этап 2 описан как единый этап:
- Backend (REST API + бизнес-логика)

Для текущей реализации этот этап **разделён на два подэтапа**:

- **Этап 2.1** — DAO + Service слой + тесты
- **Этап 2.2** — REST API / Controller слой

При выполнении текущей задачи нужно ориентироваться на `prompts/global.md`, но считать, что:
- REST API и controller слой **не входят** в текущий scope
- на этом этапе реализуется только backend core без HTTP-слоя

## Цель

Реализовать backend core будущего backend-сервиса:

- доменные объекты
- DAO-слой через Prisma
- service слой с бизнес-логикой
- тесты для service слоя

На данном этапе не нужно поднимать HTTP API.  
Главная цель — получить работающую и покрытую тестами бизнес-логику.

## Что должно быть реализовано

### 1. Monorepo-структура

Нужно создать отдельное приложение backend в monorepo.

Ожидаемая идея структуры:
- `apps/backend/` — отдельный NestJS проект
- внутри него собственный `Dockerfile`
- Prisma относится к backend-приложению

Если структура уже существует, нужно аккуратно встроиться в неё.

---

### 2. DAO-слой

В этом проекте Prisma считается DAO-слоем.

Нужно:
- подключить Prisma к backend-приложению
- организовать доступ к данным через DAO-слой
- не смешивать бизнес-логику и прямые Prisma-вызовы в одном месте без структуры

Разрешается:
- использовать PrismaService
- создать DAO-классы / data access classes / adapters поверх Prisma

Важно:
- на этом этапе не нужен controller слой
- на этом этапе не нужен REST API

### 3. Доменные объекты и контракты

Нужно явно выделить и описать:

- доменные сущности / типы / enum

- интерфейсы зависимостей service слоя
- интерфейсы DAO-классов, если они используются

Важно:
- сначала определить контракты
- потом писать тесты
- потом реализацию

### 4. Service слой

Нужно реализовать service слой, покрывающий бизнес-логику проекта.

Ожидаемые use cases:

#### Users
- получение локального пользователя по `authSubject`
- создание локального пользователя
- поиск пользователя по id
- поиск пользователя по email, если это нужно для сценариев создания direct/group chat

#### Chats
- получить список чатов пользователя
- получить чат по id, если пользователь является участником
- создать direct chat
- создать group chat
- удалить group chat
- выйти из group chat
- получить список участников чата
- исключить участника из group chat

#### Messages
- получить историю сообщений чата
- отправить сообщение в чат

## Бизнес-правила

Нужно реализовать бизнес-правила в service слое:

- пользователь может читать и писать только в свои чаты
- direct chat существует только между двумя пользователями в единственном экземпляре
- direct chat нельзя удалить
- из direct chat нельзя выйти
- group chat должен иметь ровно одного `OWNER`
- создатель group chat получает роль `OWNER`
- остальные участники group chat получают роль `MEMBER`
- `OWNER` может удалить group chat
- `OWNER` может исключать участников
- `MEMBER` может выйти из group chat
- `OWNER` не может выйти из group chat
- сообщения не удаляются
- сообщения не редактируются

Если какое-то правило нельзя выразить только на уровне схемы БД, оно должно быть реализовано в service слое.

## Закладка под будущую аутентификацию

На этом этапе Keycloak **не интегрируется**, но backend core должен быть к нему подготовлен.

Нужно явно заложить:

### 1. Внешнюю identity-модель
Пользователь backend должен быть связан с внешним identity через:
- `authSubject`

Нельзя:
- добавлять `passwordHash`
- реализовывать local login/password auth

### 2. Current user abstraction
Нужно ввести прикладную абстракцию текущего пользователя, например:
- `CurrentUser`
- `AuthenticatedUser`
- `RequestUser`

Минимально:
- `authSubject`
- локальный `userId` можно определять в сервисах или отдельном mapping-слое

### 3. Отделение identity от transport
Service слой не должен зависеть:
- от HTTP
- от JWT
- от Nest guards
- от Keycloak SDK

Service слой должен принимать уже нормализованный прикладной объект пользователя или его идентификатор.

### 4. Подготовку к будущему middleware/guard
Нужно проектировать сервисы так, чтобы позже controller/guard/middleware могли просто передавать:
- `CurrentUser`
- либо `userId/authSubject`

Без переписывания бизнес-логики.

## Требования к тестам

Нужно качественное покрытие service слоя.

Требования:
- тесты должны запускаться локально
- тесты должны проходить без ошибок
- тесты должны быть читаемыми
- тесты должны проверять поведение, а не внутреннюю реализацию
- по возможности не использовать реальную БД в unit-тестах service слоя
- DAO-зависимости можно мокать / фейкать

Нужно протестировать как минимум:

### Chats
- создание direct chat
- запрет создания второго direct chat для той же пары
- создание group chat
- получение списка чатов только для участника
- удаление group chat владельцем
- запрет удаления group chat не-владельцем
- запрет удаления direct chat
- выход участника из group chat
- запрет выхода owner из group chat
- запрет выхода из direct chat
- исключение участника владельцем
- запрет исключения участника не-владельцем

### Messages
- отправка сообщения участником чата
- запрет отправки сообщения не-участником
- получение истории сообщений участником
- запрет чтения истории не-участником

### Users / identity
- получение или создание локального пользователя по `authSubject`, если такой сценарий реализуется
- корректная работа с `CurrentUser`
